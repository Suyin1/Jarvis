### ⚙️ MCP 服务的底层：输入输出是怎么交互的？

在了解具体操作前，我们先快速看一眼 MCP 服务的交互机制，这有助于理解整个过程。

MCP（模型上下文协议）是一个标准化的协议，让AI应用（客户端）能安全地调用外部工具（服务器）。它底层使用 `JSON-RPC 2.0` 作为消息格式进行通信。这意味着，AI客户端和MCP服务器之间的所有交互，都是通过JSON格式的“请求”和“响应”来完成的。

*   **输入**：AI助手根据你的自然语言指令（例如“检查`readme.md`文件的写作规范”），向`vale-mcp-server`发送一个标准化的JSON请求，其中包含了要执行的工具名和参数。
*   **输出**：`vale-mcp-server` 执行Vale引擎后，同样会返回一个标准的JSON响应，包含了检查结果。

在实际使用中，你的同事无需关心底层的JSON格式，只需要通过自然语言发出指令即可。

### ✍️ 第一步：实现“规范检查” (以 `vale-mcp-server` 为例)

我们先来看看如何实现你所说的“规范检查”功能。以 `vale-mcp-server` 为例，它就像是团队内部的一个“自动文档评审员”。

1.  **启动与配置**：你的同事在他们的AI客户端（如VS Code、Claude Desktop）中启动 `vale-mcp-server`。这个服务器会自动读取项目根目录下的 `.vale.ini` 配置文件，这意味着你团队的所有文档规范都可以通过这个文件来统一。

2.  **发起检查请求**：当同事在AI助手里输入“请检查我的`docs/guide.md`文件，看看哪里不符合规范”，背后的流程大致如下：

    *   **AI客户端** 会向 `vale-mcp-server` 发送一个类似这样的JSON-RPC请求：
        ```json
        {
          "jsonrpc": "2.0",
          "id": 1,
          "method": "tools/call",
          "params": {
            "name": "check_file",
            "arguments": {
              "file_path": "docs/guide.md"
            }
          }
        }
        ```
        > **注意**：`vale-mcp-server` 提供的核心工具名就是 `check_file`。

3.  **获取检查报告**：`vale-mcp-server` 执行Vale引擎，并将检查结果包装在一个JSON响应中返回给AI客户端。虽然你同事看到的是AI助手将结果友好地展示出来，但底层的结构化数据大概是这样的：
    ```json
    {
      "jsonrpc": "2.0",
      "id": 1,
      "result": {
        "content": [
          {
            "type": "text",
            "text": "## Vale Check Report for docs/guide.md\n\n**Line 15**: [warning] Consider rephrasing passive voice: 'The function is called.' -> 'Call the function.' (Microsoft.Passive)"
          }
        ]
      }
    }
    ```
    > 这只是简化版，实际返回内容会更丰富，可能会按严重性（error/warning/suggestion）对问题进行分类。

### 🛠️ 第二步：实现“自动修改” (组合使用 `vale-mcp-server` 和 `mcp-file-editor`)

接下来是关键的一步，也就是你提到的“按规范自动修改”。这需要结合一个能够操作文件的MCP服务器，比如 `mcp-file-editor`。它提供了丰富的文件操作能力，能让我们精准地修改文档。

*   **工作原理**：AI助手可以扮演“决策者”的角色，它先调用 `vale-mcp-server` 获取需要修改的地方，然后自己制定修改策略，最后调用 `mcp-file-editor` 提供的工具来精准修改文件。

*   **支持的操作**：`mcp-file-editor` 提供了多种精确操作文件的工具，例如：
    *   `replace_in_file`：基于正则表达式替换特定内容。
    *   `replace_lines_in_file`：替换指定的连续多行。
    *   `insert_into_file`：在文件中的指定位置（如某行之前/之后）插入新内容。

*   **完整流程示例**：整个“自动修改”的流程如下：

| 步骤 | 参与者 | 行为描述 | 底层工具调用（示例） |
| :--- | :--- | :--- | :--- |
| **1. 表达诉求** | 你的同事 | 在AI助手中输入：“请帮我修改`docs/api.md`，确保所有 API 端点描述都以动词开头。” | (自然语言指令) |
| **2. 检查定位** | AI助手 + `vale-mcp-server` | AI助手将文件交给`vale-mcp-server`，获得一份包含问题行号和建议的检查报告。 | `tools/call` with `check_file` |
| **3. 生成方案** | AI助手 | AI分析报告，理解“需要将以名词开头的描述改为以动词开头”，并生成具体的修改方案。 | (AI的推理过程) |
| **4. 执行修改** | AI助手 + `mcp-file-editor` | AI将修改方案转化为一系列精准的文件操作指令。 | `tools/call` with `replace_in_file`, `insert_into_file` 等 |
| **5. 完成反馈** | AI助手 | 助手向你报告修改完成，你可以在IDE中直接看到更新后的文件。 | (完成提示) |

### 🛠️ 1. 能否将两个MCP服务合二为一？

**完全可以，而且这正是MCP协议推荐的做法**。一个MCP服务本质上是一个工具箱，你可以将检查（Vale）和修改（File Editor）这两个能力，作为两个独立的“工具”注册在同一个服务里，无需维护两个独立的服务进程。AI客户端会通过统一的接口，根据意图自动选择调用哪个工具。

*   **技术实现**：难度不高。核心是区分“工具”（Tools），一个用于检查，一个用于修改。
    *   **Check Tool**: 内部调用 `vale` 命令行工具，解析其JSON输出。
    *   **Edit Tool**: 实现文件读写、内容替换等操作。
*   **开发方式**：自己开发的成本很低。以Go语言为例，使用`mark3labs/mcp-go`这样的库，核心代码可能不到100行。
*   **现成方案**：如果不想自己写，也有如`mcpbundler`这类现成的“聚合器”项目，能帮你将两个独立的MCP服务合并为一个。

### 📂 2. Vale-MCP-Server如何读取用户本地修改？

**是的，它能读取，但必须在用户（客户端）主动授权和配置的范围内。**

*   **关键机制：“Roots”（工作区根目录）**：这是MCP协议的核心安全机制。用户必须在客户端（如VS Code）配置中，明确声明允许MCP服务访问哪些目录。
*   **运作流程：**
    1.  **客户端声明**：MCP客户端启动时，会向服务器发送一个能力声明，表明自己支持并管理 `roots`。
    2.  **服务器请求**：当需要访问文件时，服务器会向客户端发送 `roots/list` 请求，询问当前可访问的根目录列表。
    3.  **客户端授权**：客户端收到请求后，会弹窗请求用户确认，或根据预设配置，自动返回已授权的目录列表。
    4.  **服务器操作**：`vale-mcp-server` 拿到目录后，才能在这些目录下执行 `vale` 命令，检查你的文件。

简单来说，整个文件访问的“门禁卡”始终掌握在客户端（用户）手中，未经用户授权，服务器无法读取任何文件。

### ✍️ 3. mcp-file-editor如何直接操作本地文件？

**和读取类似，`mcp-file-editor` 也能操作本地文件，且同样受`roots`机制约束。**

它是一个实现了MCP协议的“文件系统服务器”，工作原理如下：

1.  **定义操作**：它将文件的增删改查等操作，封装成一系列标准的“工具”，如`read_file`、`write_file`、`list_directory`等，并对外暴露。
2.  **接收指令**：当你发出修改文件的指令时，AI助手会调用`mcp-file-editor`对应的工具。
3.  **安全执行**：服务器收到指令后，会先验证目标文件是否在客户端的`roots`范围内。只有验证通过，才会执行操作，否则拒绝执行。
4.  **结果返回**：操作执行后，服务器将结果（成功或失败）返回给AI客户端。

整个修改过程，同样需要在客户端明确授权的目录范围内进行。

### ⚠️ 重要补充：潜在的安全风险

需要特别提醒，像`mcp-editor`这类第三方文件操作MCP服务，部分版本是**缺乏内置的严格权限控制的，完全依赖客户端的批准机制**。此外，MCP服务配置信息（如凭证）在用户主目录的明文存储，也可能带来潜在风险。

因此，在团队内部使用时，务必通过配置`roots`来严格限制访问范围，并对关键修改操作启用客户端审批，不能完全依赖服务器自身的安全机制。

3.  **部署与测试**：编译并配置到AI客户端，进行充分的功能与安全测试。

这样，你就能得到一个既符合团队需求，又高度可控的二合一MCP服务器了。



这两个工具的安全性可以分开看：核心的Vale非常安全，而MCP Go SDK在使用时需要多加留意，尤其是在选择通信协议上。

### 🛡️ 核心工具Vale：本地运行，风险极低

Vale本身是一个完全在本地运行的命令行工具，它不会将你的文档内容发送到任何远程服务器，因此没有数据外泄的风险。网络上常见的关于“Vale”的安全漏洞，通常是指另一款名为“ValeApp”的移动应用程序，与我们在讨论的文档检查工具Vale没有关系，可以直接忽略。

安全使用Vale，关键在于管理好它可能“看到”的数据。作为最佳实践，建议：
*   使用 `.gitignore` 文件将 `.vale.ini` 配置文件和内部词汇表排除在版本控制之外，防止意外上传。
*   避免在Vale的规则文件中硬编码任何敏感信息，如内部术语或项目代号。

### ⚠️ MCP Go SDK：需注意潜在风险

MCP Go SDK的安全性关键在于通信方式的选择。对于在内网团队间共享的服务，风险主要源于对HTTP传输方式的配置不当，具体体现在以下两点：

1.  **DNS重绑定攻击 (CVE-2026-34742 / CVE-2026-34742)**：如果将服务器部署为HTTP服务，默认情况下可能未启用DNS重绑定保护。在特定条件下，恶意网站可能绕过安全策略，向你的本地MCP服务发送请求。
2.  **跨站请求伪造 (CVE-2026-33252)**：同样地，若HTTP服务器未正确校验请求的来源（`Origin`头），攻击者可能构造请求，在你不知情的情况下调用MCP工具。

不过，这些问题都有明确的修复方案和规避措施，可以通过采用更安全的部署方式来避免。

### 🛠️ 内网安全部署最佳实践

要安全地在公司内网部署这个服务，可以参考以下“三步走”的策略：

#### 1. 第一步：首选Stdio传输，彻底规避风险
在AI客户端（如VS Code插件）的配置中，将MCP服务的通信方式指定为`stdio`（标准输入输出），这是最安全的方式。
```json
{
  "mcpServers": {
    "vale-editor": {
      "command": "/path/to/your/vale-editor-mcp-server",
      "args": ["/path/to/allowed/project/dir"]
    }
  }
}
```
这样做的好处是，服务仅在你本机运行，与AI客户端通过进程内的标准输入输出通信，完全不监听任何网络端口，从根源上杜绝了上述所有网络攻击的风险。

#### 2. 第二步：如用HTTP，务必更新并加固
如果因特殊原因必须使用HTTP传输（例如需要集中部署），请务必采取以下措施：
*   **更新SDK至最新版本**：确保使用的`go-sdk`版本 **≥ v1.4.1**，该版本已修复了上述所有已知的CVE漏洞。
*   **启用访问控制与身份验证**：为MCP服务器配置身份验证，并限制其监听的IP为`127.0.0.1`（仅本地访问），避免对外暴露。

#### 3. 第三步：离线部署，确保供应链安全
在公司内网环境，从源头保证软件的纯净至关重要。你可以参考下面的表格，确认各组件的离线部署方法。

| 组件 | 离线部署方法 | 关键点/注意事项 |
| :--- | :--- | :--- |
| **Vale** | 1. 从GitHub官方Release页下载对应平台的二进制文件。<br>2. 通过内部安全渠道分发。 | 官方不提供内网镜像，需手动下载和分发。 |
| **MCP Go SDK** | 1. 在联网机器上使用`go mod vendor`将依赖下载到项目的`vendor/`目录。<br>2. 将包含`vendor/`目录的整个项目打包传入内网。<br>3. 在内网使用`go build -mod=vendor`进行编译。 | 所有依赖被固化，确保构建的一致性和安全性。 |

### 💎 总结
总的来说，你设计的“Vale + MCP”方案在公司内网使用是完全可行的，并且安全可控。核心结论如下：

*   **Vale是安全的基石**：它本身没有数据外传风险，只需注意内部配置文件的保密。
*   **MCP安全可控**：通过选择正确的传输方式可以轻松规避风险。
    *   **首选方案**：在AI客户端配置中使用`stdio`通信。
    *   **备选方案**：如必须使用HTTP，则务必**更新SDK至最新版本**并**添加身份验证**。
*   **离线部署**：参考上述指南，可完全实现内网离线部署，保障供应链安全。
